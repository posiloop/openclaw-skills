# Intelligent Customer Service

## Overview

使用這個 skill 建立一個可持續維護的智能客服工作流，讓代理能在嚴格受控的知識範圍內回答問題，並把新增規則與知識來源整理成本地檔案。

此 skill 只有兩個正式模式：問答模式、開發模式。執行前先辨識目前屬於哪一種，避免混用。

## Workflow

1. 先判斷使用者要進入問答模式還是開發模式。
2. 讀取本 skill 的本地知識檔。
3. 依模式執行對應流程。
4. 若開發模式有新增或修改內容，更新本地 Markdown 檔。
5. 回覆時明確遵守模式限制，不自行腦補知識庫外內容。

## Mode 1: 問答模式

### Goal

根據既有 FAQ、規則與知識庫回答問題。

### Rules

- 只能根據本 skill 已保存的知識內容回答。
- 不得使用模型常識補足知識庫缺口。
- 若問題無法從 FAQ、規則或知識檔中找到明確依據，直接回答：這不在我的知識庫中
- 若知識庫有相近內容但不足以支持精確回答，仍回答：這不在我的知識庫中
- 若使用者同時要求新增或修改知識，先切換為開發模式處理，不能在問答模式直接改資料。

### Answering Process

1. 讀取 `references/faq.md`、`references/rules.md`、`references/knowledge-index.md`。
2. 必要時再讀取 `references/knowledge/` 下對應主題文件。
3. 檢查是否有明確答案或規則可支撐回覆。
4. 有依據才回答，且優先採用 FAQ 與明確規則。
5. 無依據時只輸出固定句子：這不在我的知識庫中

## Mode 2: 開發模式

### Goal

接收新規則、FAQ 配對與知識素材，整理成可維護的本地 Markdown 知識庫。

### Supported Inputs

- 規則說明
- 問答配對
- 文件內容或文件檔案
- 錄音轉文字內容或音訊檔案
- 影片摘要、逐字稿或影片檔案
- 其他可轉成文字知識的來源

### Rules

- 所有新增知識都要落地成 Markdown 檔，不只停留在對話裡。
- 先整理、去重、標準化，再寫入對應檔案。
- 若來源內容有不確定、衝突或缺漏，先向使用者確認，再定稿。
- 新增資料時要盡量維持 FAQ、規則、知識文件分層，不把所有內容塞進單一檔案。
- 若收到非文字素材，先萃取或整理成文字，再寫入知識檔。

### Update Process

1. 判斷輸入屬於規則、FAQ、還是一般知識素材。
2. 先讀取既有檔案，避免重複與衝突。
3. 將內容整理成標準 Markdown。
4. 依類型寫入對應檔案：
 - FAQ 寫入 `references/faq.md`
 - 規則寫入 `references/rules.md`
 - 主題知識寫入 `references/knowledge/<topic>.md`
5. 更新 `references/knowledge-index.md`，讓後續問答模式知道有哪些主題知識。
6. 回報本次新增或修改了哪些檔案。

## File Layout

```text
references/
├── faq.md
├── rules.md
├── knowledge-index.md
└── knowledge/
 └── *.md
```
用固定格式維護 FAQ：
```# FAQ

## Q: 問題
A: 答案

## Q: 另一個問題
A: 另一個答案
```
用條列方式維護硬性規則與回答限制：
```# Rules

- 規則 1
- 規則 2
```
記錄主題知識清單與用途：
```# Knowledge Index

- product: 產品介紹、方案、功能限制
- billing: 收費、付款、退款
```
每個主題一份文件，內容以可檢索、可引用為主：
```# Topic: product

## Summary

主題摘要

## Details

- 重點 1
- 重點 2
```
- 使用者若只說「新增規則」或「把這段加入知識庫」，視為開發模式。
- 使用者若只問客服問題，視為問答模式。
- 若使用者沒有明說模式，但訊息同時包含提問與資料更新需求，先拆成兩段處理，優先完成開發模式更新，再回到問答模式。
- 回答內容要可追溯到本地知識檔。
- 若後續需要支援更細的檢索、標籤或版本控管，可以在此 skill 基礎上擴充，但不要破壞兩模式的核心規則。
