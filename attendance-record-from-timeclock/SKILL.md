# 打卡照片出勤整理

## 概要

將使用者提供的打卡機照片、紙本打卡卡照片或其他可讀取的出勤影像，整理成固定格式的「出勤紀錄」文字區塊。

重點不是回傳原始 OCR，而是產出可直接貼進每日營業表的最終格式。

## 輸出目標

輸出時一律使用以下樣式，保留分隔線與欄位名稱：

```text
出勤紀錄⋯⋯⋯⋯⋯
巧鈺（正）
出勤時間：8-16
工時：8
⋯⋯⋯⋯⋯⋯⋯⋯⋯
月娥（試）
出勤時間：8-16
工時：8
⋯⋯⋯⋯⋯⋯⋯⋯⋯
頤喬（PT）
出勤時間：假
工時：
⋯⋯⋯⋯⋯⋯⋯⋯⋯
方儀（支援）
出勤時間：8-16
工時：8
⋯⋯⋯⋯⋯⋯⋯⋯⋯⋯⋯
```

## 工作流程

### 1. 先收照片，先確認是否還有照片要補上

當使用者上傳第一張或第一批打卡照片後，先不要立刻確認年月，也不要立刻輸出結果，先問：
- 「還有沒有打卡照片要繼續上傳？」

規則：
- 若使用者表示還要繼續上傳，先持續收資料，不做最終整理。
- 在使用者尚未明確表示「沒有了」、「就這些」、「可以開始整理」之前，不進入下一步。
- 若使用者一次補上多張照片，先將它們視為同一次待整理批次，年月是否一致留待下一步確認。

### 2. 逐張、逐卡確認基本資訊

在使用者確認沒有更多照片後，必須改為逐張照片、逐張卡片確認，不可把整批混在一起看。

每張照片中的每一張打卡卡，都至少要先確認三件事：
- 打卡年月
- 打卡人姓名
- 這張卡上實際有打卡紀錄的日期範圍

先問的方式可類似：
- 「我先逐張確認。這一張照片裡的第 1 張卡看起來是 `YYYY 年 MM 月`、姓名是 `某某`、有紀錄的日期範圍是 `3/1-3/15`，這樣對嗎？」

規則：
- 無論使用者上傳幾張照片，或一張照片裡有幾張卡，都必須逐張卡片確認，不可只確認整批的總年月。
- 若某張卡的年月辨識不清楚，必須單獨標示這張卡的年月待確認，不可直接沿用其他卡片的年月。
- 若某張卡的姓名辨識不清楚，必須單獨標示這張卡的姓名待確認。
- 若某張卡只能看出部分日期，仍要先確認目前可見的日期範圍，例如 `3/16-3/31`、`僅 3/27-3/31 可見`。
- 若同一張照片中有多張卡，每張卡都要各自確認年月、姓名、日期範圍，不可合併成一句帶過。
- 若多張卡看起來屬於同一人不同半月，也要先分別確認每張卡的日期範圍，再在後續步驟合併。

### 3. 逐張分析整批照片，再統整人名與打卡時間

- 使用者可以一次上傳一批照片，但分析時必須一張一張處理。
- 先為每張照片中的每一張卡建立暫存結果，至少記錄：照片編號、卡片編號、可見年月、可見姓名、可見日期範圍、可見上下班時間、手寫工時、模糊或遮擋處。
- 不要一次要求模型完整解析整批照片。若工具支援多張，也應以「逐張、逐卡、可重試」為原則。
- 若某張照片失敗，不應導致整批中止；要保留已完成的照片結果，接著分析下一張。
- 若某張照片資訊過多或字太小，優先改成局部裁切或按單一卡片、單一日期區段辨識。
- 在完成逐張卡片的年月、姓名、日期範圍確認後，再按人員歸戶，同一人跨多張照片或多張卡片的紀錄要合併看待。
- 若同一人在不同卡片上分別記錄同月份不同日期，需先合併成同一人的月資料後再整理。
- 優先擷取每人在該月各日期的最早上班時間與最晚下班時間。
- 若照片中同一人有多筆紀錄，先判斷是否為同日上下班卡，再整理為一段出勤時間。
- 若只有單邊打卡，視為資訊不足，不要硬算完整工時，應標示待確認。

### 4. 先顯示初步偵測資料，確認上班日、非上班日與手寫內容

在使用者完成照片上傳、且年月、姓名、日期範圍逐張卡片確認完成後，就要立刻先向使用者展示本批資料的「初步偵測結果」。完成這一步並獲得使用者修正後，後續必須把本批所有已辨識到的上班日逐日掃描、整理並寫入本地端檔案，預先保存每日工時與待確認欄位，方便之後快速讀取與輸出。

初步偵測確認一律盡可能簡短，優先使用以下格式：
- `姓名 / 年月 / 上班日 / 下班日 / 手寫偵測確認`

其中：
- `上班日` = 目前偵測有上班的日期
- `下班日` = 目前偵測沒上班的日期
- `手寫偵測確認` = 只列出需要使用者確認或修正的手寫內容

規則：
- 這一步只在整批照片收集完成後做一次，作為最一開始的總體校對保險。
- 一旦這一步完成並獲得使用者修正，後續同批任務在輸出任何指定日期出勤紀錄時，應直接輸出正式版，不要再次回到初步偵測摘要。
- 內容要盡量短，不貼原始 OCR，不寫多餘解釋。
- 若某些日期無法可靠判斷是否有上班，才在 `手寫偵測確認` 補充簡短說明。
- 若手寫內容可能影響工時判定，必須放進 `手寫偵測確認` 請使用者確認。
- 使用者在這一步的修正，視為高優先級證據，後續輸出時必須保留，不得被 OCR 重跑覆蓋。
- 在這一步確認完成後，不要只等使用者指定日期才重掃。要先把本批所有「有上班」的日期逐日建立本地端紀錄，至少保存：姓名、職位、年月、日期、上班時間、下班時間、工時、手寫工時、資料來源、是否待確認。
- 若某日只能判讀部分時間，也要寫入本地端檔案，並把 `是否待確認` 標為 true，不可直接略過。
- 本地端紀錄應採容易讀取與後續追加更新的格式，例如 JSON、YAML 或 Markdown 結構化表列；同一批次後續補拍或人工修正時，應更新既有紀錄而不是另開一套互相矛盾的資料。

### 5. 統一確認所有人的姓名與職位

在初步偵測資料確認完後，先列出本批辨識到的人員，統一請使用者確認：
- 姓名
- 職位標籤（正 / 試 / PT / 支援）
- 目前辨識結果是否正確
- 是否有缺少尚未辨識到的人員
- 是否與現有常用名單一致
- 若出現新的人員或不同標記，是否要加入常用名單

規則：
- 同一位員工在同月份內可能對應多張卡片，必須先做「人員歸戶」，不可把卡片數量直接當成人數。
- 若多張卡片看起來可能屬於同一人，先暫時合併成同一候選人，並請使用者確認，不要強迫使用者逐張卡片一對一對應姓名。
- 向使用者確認時，優先用「目前辨識到的人員名單」而不是「第幾張卡是誰」的方式提問；只有在真的無法歸戶時，才補問特定卡片屬於誰。
- 輸出名稱以使用者最後確認版本為準，不自行改寫。
- 若照片辨識名稱不清楚，先用「姓名待確認」方式詢問，不直接定案。
- 必須主動詢問使用者本批是否有缺少辨識的人員，或是否有辨識錯誤需要補正。
- 若使用者已有常用名單，應比對本次辨識結果與常用名單是否一致。
- 若本次出現常用名單外的新成員，或既有人員的職位標籤與常用名單不同，必須主動詢問是否更新或加入常用名單。
- 若同一人跨多張卡片出現，後續整理指定日期與累計工時時，需合併該人所有已確認卡片資料後再計算。
- 若有固定名單中的成員在指定日期沒有出勤資料，需輸出該人並標示 `出勤時間：假`、`工時：`。

### 6. 確認要輸出哪一天的出勤紀錄

在姓名與職位都確認，且缺漏人員與辨識修正都處理完成後，再確認本次要輸出的日期。

規則：
- 若使用者已明確指定日期，直接使用。
- 若本批照片含多天資料且使用者尚未指定，必須先問要輸出哪一天。
- 同時保留本批同年月內截至該日期的所有已確認資料，因為最後還要計算累計工時。

### 7. 整理指定日期的出勤時間

在正式輸出指定日期前，必須先做「逐人逐卡複核」，避免漏掉出勤紀錄。

逐人逐卡複核規則：
- 先以本批已確認的人員名單或固定名單，逐一檢查每位員工。
- 對每位員工，反查其名下所有已確認歸戶的卡片，不可只看單一張卡或單次辨識摘要。
- 逐張確認指定日期是否有打卡紀錄、手寫工時，或其他足以判斷出勤的資訊。
- 對每位員工與指定日期，必須建立「日期級證據清單」，整合所有可用來源，例如：原始卡片、補充照片、局部補拍、使用者人工補正。
- 先用每張卡已確認的日期範圍，判斷哪些卡片理論上涵蓋指定日期。
- 只要某張卡的日期範圍涵蓋指定日期，就必須對該卡做「指定日期列級重掃」，逐列確認該日是否有打卡、手寫工時或其他出勤證據。
- 未完成指定日期列級重掃前，不得因整張卡摘要未提到該日期，就直接判定為 `假`。
- 若指定日期那一列模糊、遮擋或無法可靠辨識，應標示待確認，不可直接視為未出勤。
- 證據來源優先級依序為：使用者人工補正 > 補充照片或局部補拍 > 原始卡片明確辨識 > 不確定 OCR 推測。
- 只要較高優先級來源已確認該日有出勤，後續重跑時不得被較低優先級來源覆蓋為 `假` 或無資料。
- 若同一人在不同卡片或不同來源中出現同日資料，需先做「日期級證據合併」，再決定最終出勤時間與工時。
- 只有在某位員工所有對應卡片與所有補充來源都完成指定日期列級重掃，且都確認沒有該日期紀錄後，才能標示：`出勤時間：假`
- 若不同來源之間的時間互相衝突，先保留全部候選版本，優先採用最清楚、最完整、來源優先級最高者；仍無法定案時再向使用者確認。
- 若使用者曾手動補正某位員工某日有出勤，後續同批整理時必須優先採用該補正結果，不可再被 OCR 判讀覆蓋。
- 若使用者明確否決某位員工某日有出勤，該否決結果同樣視為最高優先級證據，後續不可再被 OCR 重新判為有上班。
- 在決定最終時間前，必須先做「日期列身份確認」，確認該列資料確實屬於該卡本人、沒有看錯相鄰列、相鄰卡片或欄位。
- 若只讀到單一時間點、單邊打卡或不符合卡片常見結構，需先做合理性檢查；不足以形成完整上下班區間時，應標示待確認，不可直接視為完整出勤。

時間輸出規則：
- 以 `上班-下班` 形式輸出，例如 `8-16`、`8:30-17:00`。
- 若分鐘為 `00`，可省略為整點，例如 `8-16`。
- 若原始時間不是整點，保留分鐘，例如 `8:25-16:40`。
- 若可明確判斷當日未出勤，輸出：`出勤時間：假`
- 若無法可靠判斷上下班完整區間，輸出可讀到的資訊並明確標註需確認，例如：`出勤時間：8-？`

### 8. 計算工時

工時計算規則：
- 預設以 `下班時間 - 上班時間` 計算，單位為小時。
- 若照片上已有手寫工時，且與打卡時間相符或打卡時間不清楚時，可優先採用手寫工時。
- 若結果為整數，直接輸出整數，例如 `8`。
- 若有 30 分鐘，輸出 `0.5` 精度，例如 `7.5`。
- 若分鐘不是 00 或 30，保留到小數點後一位，採四捨五入，例如 `7.3`。
- 若為休假或未出勤，工時留空：`工時：`
- 若資料不足無法可靠計算，工時留空，並在必要時附一句簡短提醒請使用者確認。

### 9. 計算截至指定日期的累計工時

完成指定日期出勤整理後，還要額外輸出：
- 每位員工在本批同年月資料中，截至第 5 步所指定日期為止的累計總工時
- 全部員工截至該日期的加總工時

規則：
- 只加總本批次已上傳、已確認且日期不晚於指定日期的資料，不推測未上傳日期。
- 若某日工時不確定，該筆不要硬加總，並在總結時註明有未確認項目。
- 休假不列入工時。

### 10. 最終回覆格式要求

在完成照片收集、年月確認、逐卡姓名與日期範圍確認、初步偵測資料確認、人員姓名與職位確認、缺漏與修正確認、日期確認，以及指定日期前的逐人逐卡複核後，回覆時依序提供：
1. 指定日期的出勤紀錄區塊
2. 每位員工截至該日期的目前累計總工時
3. 全部人截至該日期的加總工時

出勤紀錄區塊一律使用固定格式，不要先貼大段 OCR 原文。
若只有少數欄位不確定，先給整理好的版本，再列出需確認的人名、日期或時間。

## 判讀原則

- 先求穩，不要為了完整而捏造時間。
- 人名、上下班時間、工時三者若有任一項不可靠，明確標示不確定處。
- 若照片角度、模糊、遮擋導致辨識不穩，優先請使用者補拍，而不是硬產出錯誤結果。
- 若同一人存在多段班，除非使用者明示要加總，否則先依照片能清楚判讀的主班次整理，必要時再追問。

## 本地端資料保存

- 常用名單必須保存於本地端檔案，不只存在對話上下文。
- 建議另外維護一份固定名單檔，例如 `memory/attendance-roster.json` 或同等用途檔案，保存姓名、常用職位、別名或常見誤辨識寫法。
- 每一批打卡照片在完成姓名、手寫內容與上班日確認後，必須把所有上班日的逐日工時掃描結果寫入本地端檔案，例如 `memory/attendance-records/YYYY-MM.json`。
- 每筆每日紀錄至少包含：姓名、職位、日期、上班時間、下班時間、工時、手寫值、來源照片、信心或待確認註記、最後人工修正結果。
- 若使用者後續補正時間、姓名或職位，應直接更新本地端既有紀錄，並保留最新版本作為後續輸出依據。
- 後續再次被要求整理同月份某一天時，優先讀取本地端已保存的每日工時紀錄，再視需要回頭檢查原圖，不要每次都從零開始。

## 建議工具使用

- 使用 `image` 工具讀取打卡機照片。
- 使用者可以一次上傳整批照片，但實際辨識時應逐張呼叫或逐張整理，不要把「整批完整 OCR 成功」當成單一步驟。
- 先確認使用者是否還有照片要補上，再進行逐張卡片的年月、姓名、日期範圍確認；若任一欄位不清楚，優先向使用者確認。
- 逐張分析時，先保存每張照片、每張卡片的暫時判讀結果，再進行跨照片合併。
- 在整批照片收集完成後，先產出一版「初步偵測摘要」，列出每張卡偵測到的上班日、非上班日、手寫內容與待確認項目，交由使用者先校對；之後同批輸出指定日期時直接出正式版，不再重複這個摘要步驟。
- 若某張照片辨識失敗，先繼續處理其他照片，最後再回報哪些張數需要補拍或人工確認。
- 如需跨訊息維持固定名單，沿用當前對話已出現的人員名單與身分標籤。
- 當同一人可能出現在多張卡片時，先建立「候選人 -> 對應卡片」的暫存對照，待使用者確認後再正式合併。
- 輸出指定日期前，必須以人員名單逐一反查所有對應卡片與補充來源，先用日期範圍篩出命中卡片，再對命中卡片做指定日期列級重掃，建立指定日期的證據清單，不可直接依摘要判定為假。
- 若使用者曾手動補正某人某日的出勤結果，應保存並在同批後續輸出時優先採用，不得被後續較低優先級辨識結果覆蓋。
- 若有常用名單，需主動比對本次辨識結果，發現新人員或職位標籤差異時要詢問是否加入或更新。
- 若使用者後續提供修正名單，立即以最新版本覆蓋舊對應。
