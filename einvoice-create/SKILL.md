# 開立電子發票（B2B）

## Overview

此 skill 把「開一張 B2B 電子發票」的瀏覽器操作流程自動化。流程固定，只在一處需要人工介入：
1. 開立前讓使用者最後確認所有欄位內容（圖形驗證碼由 Claude 自己讀圖辨識）

核心資料流：
```
einvoice.nat.gov.tw 登入頁
        ↓ 自動帶入帳密 + Claude 自行辨識驗證碼
       已登入 dashboard
        ↓ 自動跳轉
  /dashboard/btb/btb207w/add（B2B 開立頁）
        ↓ 先設「發票號碼」兩個下拉（自動選號 8 碼 + 當月期別）
        ↓ 詢問 + 確認 + 自動填寫
     按下「儲存並寄送」按鈕
        ↓
     導向 https://www.google.com/
```

## Triggers

- 使用者明確說「開電子發票」、「開立電子發票」、「幫我開發票」
- 使用者貼出買方統編、品項、金額並指示要開發票
- 使用者上傳訂單資料並要求建立 B2B 電子發票

## Workflow

### 步驟 1：開啟瀏覽器並登入

1. 先呼叫 `mcp__claude-in-chrome__tabs_context_mcp` 取得目前 Chrome 分頁狀態
2. 用 `mcp__claude-in-chrome__tabs_create_mcp` 開新分頁到 `https://www.einvoice.nat.gov.tw/`
3. 在首頁找到「登入」入口點進去（多半叫「登入」、「會員登入」或 user 圖示）
4. 在登入頁：
   - 身份選擇 → **「營業人/扣繳單位」**
   - 切到「**帳號**」分頁（不是憑證、不是健保卡）
   - 統一編號欄填 `60640265`
   - 帳號欄填 `60640265`
   - 密碼欄填 `$60640265$APr`
5. **驗證碼處理（Claude 自己辨識，不要丟給使用者）**：
   - 用 `mcp__claude-in-chrome__javascript_tool` 把驗證碼 `<img alt="圖形驗證碼">` 畫到 canvas 並 `toBlob` 觸發 download，檔案會落在 `~/Downloads/captcha.png`
   - 用 `Read` 工具直接讀那張 PNG，自己看清楚字元（多半是 5 碼數字 / 英數混合）
   - 把辨識結果填進「圖形驗證碼」欄位
   - 若辨識不確定（字太糊、字元數對不上）：點「重新整理驗證碼」按鈕換一張，再讀一次。**不要把驗證碼丟回去給使用者讀**
6. 按「登入」
7. 確認登入成功（網址變成 dashboard 開頭、或頁面出現使用者統編）。若登入失敗：
   - 看是密碼錯（停下來告知使用者），還是驗證碼錯（自己換一張重讀）

### 步驟 2：跳到 B2B 開立頁

```
mcp__claude-in-chrome__navigate → https://www.einvoice.nat.gov.tw/dashboard/btb/btb207w/add
```

如果該網址需要 session 重整，先確認登入態還在。

### 步驟 2.5：先選擇發票號碼設定（必做）

在問使用者任何欄位 **之前**，先把頁面上的「發票號碼」兩個下拉選單設定好：

1. **第一個下拉**：選「**自動選號(8碼)**」
2. **第二個下拉**：選「**當月份的發票期別**」（依今天日期決定，例：4 月就選 `115年03-04月`、5 月就選 `115年05-06月`）

這兩個選單沒設好時，後續欄位（日期、買方、明細）可能不會正常啟用，所以一定要先設。設定完才繼續問使用者表頭資訊。

### 步驟 3：收集發票表頭資訊（第一階段：只問買方資料）

**重要：表頭與明細分兩階段問，不要一次全問完。**

第一階段只問下列三項，然後停下等使用者回覆：

1. **買方統一編號**（8 碼）
2. **買方名稱**
3. **買方地址**

⚠️ 此階段**不要**詢問商品明細、不要詢問開立日期。等使用者把買方資料給齊、你也填進欄位後，再進入步驟 4。

**為什麼分階段？** 一次列太多問題使用者容易漏；分階段也讓使用者能先校對買方資料是否正確再進入明細。

**開立日期**：系統強制當日，預設不用問。如果使用者主動指定日期，再依下面的日期可選性檢查處理。

**日期可選性檢查**（僅在使用者主動給日期時適用）：

把使用者給的日期填到開立日期欄位後，如果該欄位拒絕（disabled、灰色、不能選、或顯示錯誤訊息），不要強塞，直接告訴使用者：

> 「{使用者給的日期} 在系統上不能選，請改一個可選的日期再給我（一般是當月 + 前一個月內）」

等使用者給新日期再繼續。

### 步驟 4：收集交易明細（第二階段）

買方資料填完後才進到此階段。問使用者每一個品項的「品名 / 單價 / 數量」。可以一次給多項（例如「A 商品 100 元 1 個、B 商品 200 元 2 個」），用清單形式呈現。

**未稅單價換算（重要！）**：

平台填的是 **未稅單價**，使用者報給你的金額是 **含稅單價**。台灣 5% 營業稅標準算法：

```
input_price = round(actual_price ÷ 1.05)   # 四捨五入到整數
```

例：
- 使用者說 100 元 → 100 ÷ 1.05 = 95.2381 → 填 **95** 元
- 使用者說 350 元 → 350 ÷ 1.05 = 333.3333 → 填 **333** 元
- 使用者說 1,000 元 → 1,000 ÷ 1.05 = 952.3810 → 填 **952** 元
- 使用者說 105 元 → 105 ÷ 1.05 = 100.0000 → 填 **100** 元

對每個品項都套用一次。在跟使用者確認時，要把 **原始含稅金額** 跟 **要填入的未稅金額** 都列出來，避免混淆。

### 步驟 5：總確認

把所有資訊整理成一張清單給使用者最後確認：

```
請確認以下發票資訊：

【表頭】
• 開立日期：YYYY-MM-DD
• 買方統編：XXXXXXXX
• 買方名稱：...
• 買方地址：...

【明細】
1. 品名 A — 含稅 100 → 未稅 95 × 1
2. 品名 B — 含稅 200 → 未稅 190 × 2

確認沒問題的話請回「確認」，我就送出。
```

務必等使用者明確說「確認」、「OK」、「沒問題」之類的同意字眼才往下走。如果使用者要修改某欄位，回到對應步驟修改後再次給完整清單確認。

### 步驟 6：填表 + 儲存並寄送

1. 把確認後的資訊填到對應的輸入框：
   - 開立日期、買方統編、買方名稱、買方地址
   - 每個品項的「品名 / 數量 / 單價」（單價是換算後的未稅價）
2. 必要時用 `mcp__claude-in-chrome__find` 或 `mcp__claude-in-chrome__form_input` 對欄位精準操作
3. 找到並按下「**儲存並寄送**」按鈕
4. 等待動作完成（看 toast 訊息、redirect、或欄位變 disabled）
5. 確認成功（出現成功訊息、發票號碼、或頁面變化）

### 步驟 7：完成後導向

```
mcp__claude-in-chrome__navigate → https://www.google.com/
```

回覆使用者：
- 開立成功 ✅
- 系統回傳的發票號碼（如有）
- 已將分頁切到 Google

## Rules

- **不要在使用者沒確認前送出**：步驟 5 是強制關卡。
- **驗證碼自己辨識**：絕對不要把驗證碼圖片丟回去給使用者讀。下載 PNG 後自己 Read 圖檔判讀，看不清楚就換一張重來。
- **未稅換算用 ÷ 1.05 四捨五入**：台灣 5% 營業稅標準算法。換算後在確認頁要兩個金額都列出（含稅 → 未稅）。
- **登入失敗時不要硬重試**：先判斷是密碼錯（停下來告知使用者）還是驗證碼錯（自己換一張重讀），不要無限重試。
- **同一張發票只開一次**：如果中途確認 / 取消，回到步驟 3-5 重新來，不要重複按儲存並寄送。
- **不要關閉分頁**：完成後是 navigate 到 Google，不是 close tab。
- **機敏資訊不外流**：密碼 / 統編都不要寫到聊天視窗回覆給使用者（已知資訊不需重述），不要寫入除本 SKILL.md 以外的任何檔案。
